### Context
* 온라인 쇼핑몰(이커머스) 운영 시 주문 및 판매 데이터를 수동으로 관리하는 비효율을 해결하기 위해 n8n 워크플로우를 설계함.
* 판매 데이터를 실시간으로 수집하여 중앙 데이터베이스(MySQL)에 저장함으로써 재고 관리 및 매출 분석의 정확도를 높이는 것이 목적임.

### Core
* n8n 워크플로우 구성 (HTTP Request -> MySQL Insert) 예시
* 데이터 처리 항목을 4개(order_id, item, price, email)로 제한하여 최적화
```json
{
  "nodes": [
    {
      "parameters": {
        "url": "https://api.ecommerce.com/v1/orders",
        "authentication": "headerAuth"
      },
      "name": "Get Orders",
      "type": "n8n-nodes-base.httpRequest",
      "typeVersion": 4.1
    },
    {
      "parameters": {
        "operation": "insert",
        "table": "sales_records",
        "columns": "order_id, item_name, price, email"
      },
      "name": "MySQL Store",
      "type": "n8n-nodes-base.mySql",
      "typeVersion": 1
    }
  ]
}
```

### Insight
* 이벤트 기반 아키텍처(Event-driven Architecture): API 폴링(Polling) 방식보다 웹훅(Webhook)을 사용하여 주문 발생 즉시 데이터를 처리하는 것이 서버 부하를 줄이고 데이터 정합성(Data Consistency)을 확보하는 데 유리함.
* 데이터 스키마(Data Schema) 설계: 다양한 플랫폼의 데이터를 하나로 통합할 때는 중복을 방지하기 위해 정규화된 테이블 구조와 유니크 키(Unique Key) 설정이 필수적임.
* 장애 복구(Disaster Recovery): 네트워크 장애 시 데이터 유실을 방지하기 위해 n8n의 'Error Trigger' 노드를 활용하여 실패한 항목을 별도의 로그 테이블에 기록하거나 재시도하는 로직이 필요함.
