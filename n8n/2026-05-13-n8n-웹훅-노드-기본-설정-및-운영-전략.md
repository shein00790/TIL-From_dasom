### Context
n8n의 웹훅(Webhook) 노드는 외부 서비스나 애플리케이션으로부터 HTTP 요청을 받아 워크플로우를 트리거하는 가장 핵심적인 도구입니다. 단순히 데이터를 받는 것을 넘어, 서로 다른 시스템 간의 실시간 통신을 가능하게 하는 브릿지 역할을 수행합니다. 이번 학습을 통해 웹훅 노드의 기본 설정 방식과 테스트 및 운영 환경의 차이점을 명확히 파악했습니다.

### Core
n8n 웹훅 노드 설정 시 핵심적으로 고려해야 할 구성 요소는 다음과 같습니다.

*   **HTTP Method**: GET, POST, PUT, DELETE 등 목적에 맞는 요청 방식을 선택합니다. 일반적으로 데이터 수신에는 POST를 사용합니다.
*   **Path**: 웹훅 URL 뒤에 붙는 고유 경로를 설정합니다. (예: `my-workflow-path`)
*   **Webhook URLs**: n8n은 두 종류의 URL을 제공합니다.
    *   **Test URL**: 개발 단계에서 데이터를 확인하고 구조를 잡을 때 사용합니다. 'Listen for event'를 클릭한 상태에서만 유효합니다.
    *   **Production URL**: 워크플로우를 활성화(Activate)한 후 실제 운영 환경에서 사용합니다.
*   **Response Mode**: 요청자에게 보낼 응답 시점을 선택합니다.
    *   `On Received`: 데이터를 받는 즉시 200 OK 응답을 보냅니다.
    *   `Last Node`: 워크플로우의 마지막 노드 결과를 응답으로 보냅니다.
    *   `When Last Node Finishes`: 모든 처리가 완료된 후 특정 응답 노드의 데이터를 전달합니다.

```json
// 예시: 웹훅으로 전달된 JSON 본문 데이터 구조
{
  "headers": {
    "host": "n8n.example.com",
    "content-type": "application/json"
  },
  "params": {},
  "query": {},
  "body": {
    "event": "user_signup",
    "user_id": "12345",
    "email": "test@example.com"
  }
}
```

### Insight
n8n 웹훅 활용 시 가장 주의해야 할 점은 **Test URL과 Production URL의 엄격한 분리**입니다. 초보 사용자가 자주 하는 실수 중 하나는 워크플로우를 활성화한 후에도 여전히 Test URL을 외부 서비스에 등록해 두어 데이터가 유실되는 경우입니다. 운영 환경으로 배포할 때는 반드시 Production URL로 변경해야 합니다. 또한, 보안을 위해 `Authentication` 설정을 통해 Basic Auth나 Header Auth를 추가하여 신뢰할 수 있는 소스로부터의 요청만 허용하는 것이 기술적으로 매우 중요합니다.
