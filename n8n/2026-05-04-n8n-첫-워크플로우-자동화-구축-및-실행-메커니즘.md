### Context
n8n을 활용하여 첫 번째 자동화 워크플로우를 구축하고 테스트를 진행했습니다. AI 에이전트(AI Agent) 노드를 포함한 구성에서 무료 버전 혹은 특정 환경에서의 성능 저하 및 동작 불안정 현상을 경험했습니다. 특히 워크플로우를 활성화(Activate)하고 실행(Execute)하는 과정에서 버전 관리와 동시 실행(Concurrency)에 대한 매커니즘을 파악하는 과정이 있었습니다.

### Core
n8n 워크플로우의 실행 방식은 크게 두 가지 모드로 나뉩니다.

*   **테스트 모드 (Manual Execution)**: 에디터에서 'Execute Workflow'를 클릭하여 실행합니다. 이는 현재 편집 중인 최신 상태(Draft)를 즉시 테스트하며, 실행 결과가 UI에 실시간으로 표시됩니다.
*   **활성화 모드 (Production Activation)**: 우측 상단의 'Active' 토글을 켭니다. 이 상태가 되어야 트리거(Webhook, Cron 등)가 발생했을 때 백그라운드에서 자동으로 워크플로우가 실행됩니다.

AI 에이전트 노드 구성 시 필수 요소는 다음과 같습니다.
*   **모델 (Model)**: LLM 제공자(OpenAI, Anthropic 등)와 연결됩니다.
*   **도구 (Tools)**: 에이전트가 수행할 수 있는 기능을 정의하며, 성능은 LLM의 도구 호출(Tool Calling) 능력에 의존합니다.
*   **메모리 (Memory)**: 대화 맥락을 유지하기 위해 'Window Buffer Memory' 등을 연결합니다.

n8n 클라우드 무료/스타터 플랜의 경우 동시 실행(Concurrency) 수 제한(약 5개)이 있어, 여러 요청이 동시에 들어올 경우 큐(Queue)에 대기하거나 실행 속도가 느려질 수 있습니다.

### Insight
n8n의 작동 방식을 분석한 결과, 단순히 코드를 수정하는 것이 아니라 '활성화(Activation)' 과정이 실 서비스 적용의 핵심임을 확인했습니다. 사용자가 언급한 '이전 버전의 퍼블리셔' 문제는 n8n의 버전 관리 시스템 때문입니다. 워크플로우를 'Active'로 설정하면 해당 시점의 버전이 프로덕션 환경에 배포됩니다. 이후 에디터에서 내용을 수정하더라도 다시 저장(Save)하고 활성화 상태를 갱신하지 않으면, 트리거에 의해 실행되는 워크플로우는 여전히 '이전 버전'의 로직을 따르게 됩니다. 

또한 AI 에이전트의 동작이 원활하지 않은 것은 n8n 자체의 사양보다는 연결된 LLM의 API 응답 속도나 도구 호출(Tool Calling) 시의 타임아웃 설정 문제일 가능성이 높습니다. 따라서 효율적인 자동화를 위해서는 워크플로우의 상태(Active/Inactive)를 명확히 관리하고, 에이전트가 사용하는 도구의 복잡도를 최적화해야 합니다.
