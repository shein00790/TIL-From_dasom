### Context
디스코드 메시지를 입력으로 받아 AI 에이전트가 이를 구조화된 TIL(Today I Learned) 문서로 변환하고, GitHub 저장소에 자동으로 업로드 및 README 인덱스까지 업데이트하는 워크플로우를 구축했습니다. 파편화된 메모를 수동으로 정리하고 커밋하는 반복적인 과정을 자동화하여 지식 관리의 효율성을 높이는 것이 목적입니다.

### Core
이 워크플로우는 n8n의 다양한 노드를 유기적으로 연결하여 구성되었습니다.

*   **Discord Trigger**: 특정 채널의 `Message Created` 이벤트를 감지하여 사용자의 메모 텍스트를 트리거 데이터로 수신합니다.
*   **AI Agent Node**: 수신된 텍스트를 LLM(Claude 또는 GPT-4o)에 전달하여 미리 정의된 TIL 마크다운 포맷으로 변환합니다. 이때 시스템 프롬프트를 통해 'Context', 'Core', 'Insight' 구조를 유지하도록 강제합니다.
*   **GitHub Node (Push File)**: `File` 리소스와 `Create` 작업을 사용하여 변환된 내용을 `YYYY-MM-DD-제목.md` 형식의 파일명으로 저장소에 푸시합니다.
*   **GitHub Node (Update Index)**:
    *   기존 `README.md`의 내용을 `Get Content`로 읽어옵니다.
    *   새로 생성된 파일 경로를 불렛 포인트(`*`) 목록에 추가하는 로직을 수행합니다.
    *   `Update Content` 작업을 통해 수정된 내용을 다시 푸시하며, 이때 파일 업데이트를 위해 필수적인 `SHA` 값을 이전 노드에서 참조합니다.
*   **Discord Node (Response)**: 모든 프로세스가 완료되면 디스코드 채널로 작업 성공 메시지와 함께 GitHub 파일 링크를 전송합니다.

### Insight
n8n을 처음 사용하며 얻은 가장 큰 교훈은 각 노드의 **입력(Input)과 출력(Output)**에 대한 데이터 구조를 정확히 파악하는 것이 워크플로우 설계의 핵심이라는 점입니다. 각 노드가 내보내는 JSON 객체의 필드명을 이해하고 있으면, `Expression`을 통해 복잡한 데이터 매핑도 수월하게 처리할 수 있습니다. 특히 GitHub API를 다룰 때 파일 업데이트 시 이전 상태의 `sha` 값을 함께 전달해야 한다는 기술적 요구사항을 이해하는 것이 중요했습니다.
