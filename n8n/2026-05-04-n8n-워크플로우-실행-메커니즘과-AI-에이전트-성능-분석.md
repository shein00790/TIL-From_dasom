### Context
n8n 플랫폼을 처음 사용하여 자동화 워크플로우를 구축하고 AI 에이전트(AI Agent) 노드를 테스트했습니다. 무료 클라우드 환경에서 AI 에이전트의 응답 속도 및 안정성 이슈를 확인했으며, 편집기 상의 '실행(Execute)' 버튼과 워크플로우의 '활성화(Active)' 상태 간의 동작 차이로 인한 실행 메커니즘을 학습했습니다.

### Core
n8n의 워크플로우 실행은 크게 두 가지 모드로 구분되며, AI 에이전트 구현 시 다음 요소를 고려해야 합니다.

* **수동 실행 (Manual Execution)**: 편집기 하단의 'Execute Workflow' 버튼을 클릭하여 실행합니다. 현재 캔버스에 배치된 노드의 설정을 즉시 테스트할 수 있으며, 실행 이력은 'Executions' 탭에 별도로 기록됩니다.
* **프로덕션 실행 (Active Workflow)**: 우측 상단의 토글을 'Active'로 변경하면 트리거(Webhook, Schedule 등)에 의해 자동으로 실행됩니다. 이때는 저장된 최종 버전이 실행되므로, 편집 중인 내용이 즉시 반영되지 않을 수 있습니다.
* **AI Agent 노드 구성**:
  * 모델: GPT-4o-mini 또는 Claude 3.5 Sonnet (리소스 최적화 용도)
  * 메모리: Window Buffer Memory 또는 Postgres Chat Memory (대화 맥락 유지)
  * 도구(Tools): 계산기, Google Search, Custom HTTP Request 등 에이전트가 사용할 수 있는 도구 연결

### Insight
n8n 무료 버전(Cloud Desktop 또는 Starter)은 CPU 및 메모리 리소스 할당량이 적어 복잡한 AI 에이전트 루프나 다량의 데이터 처리에 한계가 있을 수 있습니다. 특히 사용자가 느낀 '이전 버전과의 충돌' 느낌은 수동 실행 시점과 활성화된 프로덕션 버전 간의 '저장 상태 차이'에서 기인했을 가능성이 높습니다. 워크플로우를 수정할 때는 반드시 저장(Save) 후 활성화 상태를 갱신해야 일관된 동작을 보장할 수 있습니다. 성능 개선을 위해서는 가급적 셀프 호스팅(Self-hosted) 환경으로 전환하여 서버 자원을 충분히 할당하는 것이 권장됩니다.
